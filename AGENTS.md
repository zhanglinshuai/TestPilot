# TestPilot 产品文档

## 1. 项目概述

**TestPilot** 是一个面向测试开发场景的 AI 辅助自动化测试与质量分析平台。

项目以接口自动化、UI 自动化、CI/CD 测试执行、测试报告和质量看板为基础能力，在关键分析环节引入 AI，用于辅助失败归因、测试报告总结、接口用例生成和缺陷单草稿生成。

TestPilot 的核心定位不是“让 AI 代替测试”，而是用确定性的自动化测试框架保证执行可信，用 AI 提升测试设计、问题分析和质量反馈效率。

## 2. 项目背景

在真实研发流程中，测试开发通常会面临以下问题：

- 自动化测试脚本分散，缺少统一管理平台。
- 测试执行结果难以沉淀，历史趋势和质量风险不清晰。
- CI/CD 流水线虽然能自动执行测试，但失败后的定位仍高度依赖人工。
- 接口文档变化后，测试用例补充不及时。
- 测试报告通常只展示通过率和失败日志，缺少面向研发、测试、管理者的总结信息。

随着大模型能力成熟，AI 更适合放在“辅助分析”和“辅助生成”环节，而不是直接接管测试执行。因此，TestPilot 将 AI 嵌入测试流程中最耗费人工判断的部分，提升测试开发的工作效率。

## 3. 项目目标

### 3.1 核心目标

- 搭建一个完整的自动化测试平台，支持接口测试和 UI 测试统一管理与执行。
- 支持测试任务调度、测试结果存储、测试报告展示和质量趋势分析。
- 接入 CI/CD 流水线，实现代码提交后自动触发测试。
- 引入 AI 能力，辅助失败用例分析、测试报告总结、接口用例生成和缺陷单草稿生成。
- 支持 Docker Compose 一键部署，体现工程化落地能力。

### 3.2 非目标

- 不追求让 AI 完全自动生成并执行所有测试。
- 不把平台做成通用低代码平台。
- 不优先支持复杂权限体系和多租户 SaaS 能力。
- 不在第一阶段引入过重的微服务架构。

## 4. 目标用户

### 4.1 测试开发工程师

主要使用平台管理测试用例、执行测试任务、分析失败原因、沉淀测试结果。

### 4.2 研发工程师

主要查看失败报告、接口响应、错误日志、截图和 AI 生成的初步排查建议。

### 4.3 测试负责人

主要查看项目质量看板、测试通过率趋势、失败模块分布和版本风险总结。

## 5. 核心场景

### 5.1 接口自动化测试

测试开发在平台中维护接口测试用例，配置请求方法、URL、请求头、请求体、断言规则、前后置处理和运行环境。平台执行后生成测试结果，并保存请求响应和日志信息。

### 5.2 UI 自动化测试

测试开发编写 Playwright 测试脚本，并在平台中配置执行入口。平台支持运行 UI 自动化任务，失败后保存截图、错误堆栈和执行日志。

### 5.3 CI/CD 自动触发

研发提交代码后，GitHub Actions 或 Jenkins 自动调用 TestPilot 的任务执行接口。平台执行对应测试集，并将测试结果返回流水线。严重失败时可以阻断发布。

### 5.4 失败用例 AI 分析

测试失败后，平台收集断言信息、请求参数、响应内容、错误日志、截图路径、执行环境、最近一次成功记录和 Git commit 信息。AI 根据这些上下文生成失败原因分析和排查建议。

### 5.5 Swagger 生成接口测试用例草稿

用户上传或同步 OpenAPI / Swagger 文档后，平台解析接口定义，并调用 AI 生成候选测试用例，包括正常场景、参数缺失、类型错误、边界值、鉴权失败和异常输入等。用户确认后再保存为正式用例。

### 5.6 测试报告 AI 总结

测试任务完成后，平台基于测试结果生成结构化报告，并调用 AI 输出本次执行摘要、失败模块、风险变化、重点关注问题和建议处理顺序。

### 5.7 缺陷单草稿生成

当某条失败用例被确认是产品缺陷后，平台根据日志、截图、请求响应和测试步骤生成缺陷单草稿，包含标题、复现步骤、实际结果、期望结果和环境信息。

### 功能说明

- 支持维护 UI 自动化测试脚本信息。

### 核心字段

### 功能说明

- 支持创建测试任务。
- 支持选择接口用例、UI 用例或测试套件。
- 支持手动执行、定时执行和 CI/CD 触发执行。
- 支持任务执行状态跟踪。

### 任务状态

- 待执行
- 执行中
- 执行成功
- 执行失败
- 已取消

## 6.6 测试执行引擎

### 功能说明

- 使用 Pytest 执行接口自动化测试。
- 使用 Playwright 执行 UI 自动化测试。
- 支持执行日志收集。
- 支持失败重试。
- 支持执行超时控制。
- 支持并发执行。

### 技术要点

- Pytest 负责测试调度和断言执行。
- Requests / HTTPX 负责接口请求。
- Playwright 负责 UI 自动化。
- Celery / RQ 负责任务异步调度。
- Allure 或自研报告模块负责结果展示。

## 6.7 测试报告模块

### 功能说明

- 展示测试任务整体结果。
- 展示用例通过数、失败数、跳过数和通过率。
- 展示每条失败用例的错误信息、请求响应、截图和日志。
- 支持历史报告查询。
- 支持报告导出。

### 指标示例

- 总用例数
- 通过用例数
- 失败用例数
- 跳过用例数
- 通过率
- 平均执行耗时
- 失败重试次数

## 6.8 质量看板模块

### 功能说明

- 展示项目整体质量趋势。
- 展示近 7 天、近 30 天测试通过率变化。
- 展示失败模块 Top N。
- 展示高频失败用例。
- 展示不同环境下的测试稳定性。

### 看板指标

- 测试通过率趋势
- 失败模块分布
- 用例稳定性排行
- 平均执行耗时趋势
- 缺陷关联数量
- AI 判定的失败原因分类

## 6.9 AI 失败归因模块

### 功能说明

当测试用例失败后，平台自动聚合上下文并调用大模型生成分析结果。

### 输入信息

- 用例名称
- 失败断言
- 请求 URL
- 请求参数
- 响应内容
- 错误堆栈
- 执行日志
- UI 失败截图描述
- 运行环境
- 最近一次成功记录
- Git commit 信息

### 输出信息

- 可能失败原因
- 失败类型分类
- 建议排查步骤
- 可能责任模块
- 是否建议创建缺陷
- 是否可能是脚本问题

### 失败类型分类

- 业务逻辑变更
- 接口返回结构变更
- 测试数据问题
- 测试环境问题
- 自动化脚本问题
- 前端页面元素变化
- 网络或依赖服务异常
- 未知原因

## 6.10 AI 用例生成模块

### 功能说明

平台解析 OpenAPI / Swagger 文档后，调用 AI 生成接口测试用例草稿。用户可以编辑、筛选和确认，确认后的用例进入正式用例库。

### 生成类型

- 正常请求
- 必填参数缺失
- 参数类型错误
- 字符串长度边界
- 数值边界
- 枚举值非法
- 鉴权失败
- 重复提交
- 幂等性验证
- 异常输入

## 6.11 AI 报告总结模块

### 功能说明

测试任务完成后，AI 根据测试结果生成自然语言总结。

### 输出内容

- 本次测试整体结论
- 新增失败用例
- 高频失败模块
- 与上次执行相比的变化
- 潜在发布风险
- 建议优先处理的问题

## 6.12 缺陷单草稿模块

### 功能说明

当失败用例被确认为缺陷时，AI 生成缺陷单草稿，降低测试人员整理缺陷信息的成本。

### 缺陷单字段

- 缺陷标题
- 所属项目
- 所属模块
- 严重程度
- 优先级
- 复现步骤
- 实际结果
- 期望结果
- 环境信息
- 附件说明

## 7. 系统架构

### 7.1 架构概览

```text
前端 Web
  |
  v
后端 API 服务
  |
  +--> 用户 / 项目 / 用例 / 环境 / 报告管理
  |
  +--> 测试任务调度服务
  |       |
  |       +--> Pytest 接口测试执行器
  |       +--> Playwright UI 测试执行器
  |
  +--> AI 分析服务
  |       |
  |       +--> 失败归因
  |       +--> 用例生成
  |       +--> 报告总结
  |       +--> 缺陷草稿
  |
  +--> 数据存储
          |
          +--> MySQL / PostgreSQL
          +--> Redis
          +--> 文件存储
```

| 层级 | 技术 |
| --- | --- |
| 前端 | Vue 3 / React |
| 后端 | Python FastAPI |
| 接口测试 | Pytest + Requests / HTTPX |
| UI 测试 | Playwright |
| 异步任务 | Celery / RQ |
| 数据库 | PostgreSQL / MySQL |
| 缓存 | Redis |
| 测试报告 | Allure + 自研报告页 |
| AI 接入 | OpenAI API / DeepSeek API / 通义千问 API |
| CI/CD | GitHub Actions / Jenkins |
| 部署 | Docker Compose |

## 12. 开发里程碑

### 第一阶段：自动化测试框架

目标：完成基础测试执行能力。

- 搭建 FastAPI 后端项目。
- 搭建 Pytest 接口测试框架。
- 支持配置文件管理测试环境。
- 支持接口请求、断言、日志和报告。
- 接入 Allure 报告。

### 第二阶段：测试平台 MVP

目标：完成平台化管理能力。

- 实现用户登录。
- 实现项目管理。
- 实现环境管理。
- 实现接口用例管理。
- 实现测试任务创建和执行。
- 实现测试结果入库。

目标：提升测试覆盖面和工程化能力。

## 14. 风险与应对

| 风险 | 应对方案 |
| --- | --- |
| AI 输出不稳定 | 使用结构化 Prompt，限制输出格式，保留人工确认流程 |
| AI 编造原因 | 要求基于日志证据分析，证据不足时输出待补充信息 |
| UI 自动化不稳定 | 使用合理等待、定位器规范、失败重试和截图分析 |
| 测试数据污染 | 增加前置数据准备和后置数据清理 |
| 平台功能过大 | 按里程碑开发，先完成 MVP，再增加 AI 能力 |
| CI/CD 接入复杂 | 优先支持 GitHub Actions，再扩展 Jenkins |
